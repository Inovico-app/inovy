## 1. Purpose

This procedure defines how Inovy creates, reviews, approves, distributes, maintains, and controls documented information required by the ISMS. It fulfils the requirements of ISO 27001:2022 Clause 7.5 (Documented Information) and ensures that ISMS documents are fit for purpose, properly authorized, and available to those who need them.

---

## 2. Scope

This procedure applies to all documented information within the ISMS scope (ISMS-03), including:

- ISMS foundation documents (ISMS-01 through ISMS-14)
- Information security policies (POL-01 through POL-20)
- Templates (TPL-01 and onwards)
- Records generated by ISMS processes (risk register, audit reports, corrective action records, management review minutes, training records)
- Procedures and work instructions referenced by ISMS policies

---

## 3. Document Identification Scheme

All controlled ISMS documents are assigned a unique identifier to enable unambiguous reference:

| Prefix        | Document Type                                   | Example     |
| ------------- | ----------------------------------------------- | ----------- |
| `ISMS-XX`     | ISMS foundation documents (numbered 01–14+)     | ISMS-07     |
| `POL-XX`      | Information security policies (numbered 01–20+) | POL-12      |
| `TPL-XX`      | Templates (numbered 01 onwards)                 | TPL-04      |
| `AUD-YYYY-XX` | Audit reports (year + sequence)                 | AUD-2026-01 |
| `CAR-YYYY-XX` | Corrective action records (year + sequence)     | CAR-2026-03 |
| `MGR-YYYY-XX` | Management review records (year + sequence)     | MGR-2026-01 |
| `INC-YYYY-XX` | Incident records (year + sequence)              | INC-2026-01 |

Document IDs are assigned sequentially by the ISM and are never reused, even if the document is retired.

## 5. Document Classification

All ISMS documents are assigned a classification level that governs access and handling:

| Classification   | Definition                               | Access        | Examples                                                           |
| ---------------- | ---------------------------------------- | ------------- | ------------------------------------------------------------------ |
| **Public**       | Safe for unrestricted distribution       | Anyone        | Privacy policy, security certifications                            |
| **Internal**     | For Inovy staff and contractors only     | All staff     | ISMS-XX documents, policies                                        |
| **Confidential** | Restricted to relevant team members      | Named roles   | Risk register, audit reports, incident records                     |
| **Restricted**   | Highest sensitivity; strict need-to-know | ISM + CEO/CTO | Corrective action records with sensitive findings, pentest reports |

Classification is assigned by the document owner and is shown in the document header. Changes to classification require ISM approval.

## 6. Version Control

### 6.1 Version Numbering (Semantic Versioning)

ISMS documents use a two-part semantic version number:

- **Major version (X):** Incremented for substantive changes to scope, requirements, or controls (e.g., 1.0 → 2.0)
- **Minor version (Y):** Incremented for clarifications, additions, or non-substantive updates (e.g., 1.0 → 1.1)

Version 1.0 represents the initial approved publication. Draft documents in review carry the version of the previous approved release suffixed with `-draft` (e.g., `1.0-draft`).

### 6.2 Git-Based Version Control

All ISMS documents are stored in the private GitHub repository (`inovy` monorepo) under `docs/iso27001/`. Git provides the definitive version history:

- Every change is recorded as a Git commit with author, timestamp, and commit message
- The full history of every document change is auditable via `git log`
- Previous versions are accessible via Git history; no separate version archive is maintained
- Commit messages follow the convention: `docs(iso27001): [action] [document name] ([Clause reference])`

The `main` branch always contains the current approved versions of all documents. Documents undergoing review and approval are worked on in feature branches.

## 7. Approval Workflow

All new or materially changed controlled documents follow this four-stage approval workflow:

### Stage 1: Draft

1. Document owner (typically ISM or Engineering Lead) creates or substantially revises the document in a feature branch
2. Document is given status `-draft` and its version is pre-release
3. Draft is internally self-reviewed by the author for completeness and accuracy

### Stage 2: Review

1. A pull request (PR) is raised from the feature branch to `main`
2. PR is assigned to the designated reviewer(s):
   - For ISMS-XX documents: CEO/CTO reviews and Engineering Lead is consulted
   - For POL-XX documents: ISM reviews; Engineering Lead consulted for technical policies
   - For TPL-XX templates: ISM reviews
3. Reviewers provide comments within 5 business days
4. Author addresses all comments; discussion is recorded in the PR
5. All review comments must be resolved before progressing to approval

### Stage 3: Approve

1. Approver (CEO/CTO for ISMS-XX; ISM for POL-XX and TPL-XX) provides formal approval by:
   - Adding a review approval in the GitHub PR
   - Or, for highly sensitive documents, providing approval in writing (email) recorded in the PR comments
2. The document header is updated: draft suffix removed, version finalized, effective date set
3. PR is merged to `main` by the ISM

### Stage 4: Published

1. Document is now live on the `main` branch and considered the current approved version
2. ISM communicates material updates to affected stakeholders (team members, auditors) via the designated internal channel
3. For major policy changes, ISM confirms acknowledgement by all affected staff within 14 days

## 8. Document Retention

### 8.1 Minimum Retention Periods

| Document Category                  | Minimum Retention          | Rationale                                      |
| ---------------------------------- | -------------------------- | ---------------------------------------------- |
| ISMS-XX foundation documents       | 3 years after supersession | ISO 27001 audit evidence; GDPR accountability  |
| POL-XX policies                    | 3 years after supersession | Compliance evidence                            |
| Risk register (ISMS-09)            | 3 years (each version)     | Demonstrates due diligence over time           |
| Audit reports (AUD-XX)             | 5 years                    | ISO 27001 certification cycles; legal evidence |
| Corrective action records (CAR-XX) | 5 years                    | Demonstrates continual improvement             |
| Management review records (MGR-XX) | 5 years                    | ISO 27001 Clause 9.3 evidence                  |
| Incident records (INC-XX)          | 5 years                    | GDPR breach records; legal evidence            |
| Training records                   | 5 years                    | Clause 7.2 competence evidence                 |
| Access review records              | 3 years                    | A.5.18 access review evidence                  |

### 8.2 Retention Mechanism

Git history provides an immutable retention record for all ISMS documents. Records generated by processes (audit reports, incident records, CARs) are stored in the `docs/iso27001/records/` directory with clear naming conventions and are not deleted.

Retained documents past their retention period are archived to a designated storage location (defined by ISM) and not permanently deleted without ISM and CEO/CTO approval.

---

## 9. Access Controls

- The ISMS documentation repository is a **private GitHub repository**. Access is governed by GitHub RBAC (organization membership and repository permissions).
- All Inovy engineering team members have read access to the ISMS documentation repository.
- Write access (push to feature branches, raise PRs) is available to all team members.
- Merge to `main` (approval of documents) is restricted to the ISM and CTO via GitHub branch protection rules (required approvals + restricted merge).
- Confidential and Restricted documents may be stored in a dedicated restricted-access directory or sub-repository at the ISM's discretion.

---

## 10. Annual Review Cycle

The ISM conducts an annual review of all controlled ISMS documents to ensure continued relevance, accuracy, and alignment with:

- The current state of Inovy's systems and infrastructure
- Changes to ISO 27001:2022 interpretive guidance
- Regulatory changes (GDPR, NIS2, AI Act)
- Findings from internal audits (ISMS-12) and management reviews (ISMS-13)

The annual review is scheduled in Q1 each calendar year. The review date in each document header is updated upon confirmation that the document remains current. Documents requiring substantive update follow the full approval workflow (Section 7).

Documents not reviewed within 18 months of their effective date are flagged as overdue by the ISM and escalated to the management review.

---

## 11. Document Control and External Documents

External documents referenced by the ISMS (e.g., ISO/IEC 27001:2022 standard, ISO/IEC 27002:2022 guidance, regulatory texts) are identified by their official reference in the ISMS document that cites them. The ISM monitors for updates to referenced external standards and updates ISMS documentation accordingly.

## 12. Document Control

This procedure is self-referencing and is therefore managed in accordance with its own requirements. Amendments follow the approval workflow in Section 7.
